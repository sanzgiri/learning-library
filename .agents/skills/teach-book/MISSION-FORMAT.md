# MISSION.md Format

`books/<id>/MISSION.md` captures the _reason_ this book is in the library. Every downstream decision — which concepts make the cut, how deep a lesson goes, what counts as mastery — should trace back to this document.

## Template

```md
# Mission: {Book or author}

## Why
{1-3 sentences. The concrete reason this book earns a place in the library. What should a learner be able to do after this course that they couldn't before?}

## Success looks like
- {A specific, observable thing a learner will be able to do}
- {Another specific thing}
- {…}

## Constraints
- {Anything that bounds scope: how comprehensive a first pass should be, what's explicitly deferred}

## Out of scope
- {Adjacent topics this course explicitly does not chase right now}
```

## Rules

- **One mission per book.** A course spanning several physical books by one author (an "Incerto," a trilogy) is still one mission if it's taught as one unit.
- **Concrete over abstract.** "Can redesign a real commitment to be convex" beats "understands optionality."
- **Push back on vagueness.** If the why can't be stated concretely, interview before writing anything.
- **Revise when reality shifts.** If the concept audit or the user's own priorities change what this course is for, update this file and say so — don't leave a stale mission steering lesson choices.
- **Keep it short.** If it runs past a screen, it's stopped being a compass.
